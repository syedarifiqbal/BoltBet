# BoltBet

A high-concurrency real-time sports betting engine. Built to demonstrate Staff/Senior-level backend engineering — financial data integrity, async settlement via RabbitMQ, real-time odds via Socket.io, circuit breakers, idempotency, and a full observability stack.

> **Status: Phase 1 complete.** Full Docker infrastructure is live — Postgres primary/replica, PgBouncer, Redis (persistent + ephemeral), RabbitMQ, and OpenResty API gateway with TLS, Lua-based rate limiting, and JWT blacklist at the edge.

---

## Quickstart

```bash
# Prerequisites: Docker Desktop, Node.js >=20.0.0, pnpm >=9.0.0

pnpm install

# Generate self-signed TLS certificates (once)
make certs

# Start all infrastructure
make up

# Start the app
pnpm start:dev
```

| URL | Service |
|---|---|
| `https://localhost` | API Gateway (OpenResty) |
| `http://localhost:15672` | RabbitMQ Management UI (guest / guest) |
| `localhost:6432` | PgBouncer → Postgres primary (writes) |
| `localhost:5433` | Postgres replica (reads) |
| `localhost:6379` | Redis persistent (idempotency, JWT blacklist) |
| `localhost:6380` | Redis ephemeral (rate limits, odds cache) |

Environment variables: copy `.env.example` to `.env`.

## Key Design Decisions

**Why RabbitMQ for settlement?**
When a match ends, up to 100,000 bets need payout calculation. This cannot block the HTTP request cycle. RabbitMQ decouples bet acceptance from settlement — users get an instant `202 Accepted`, workers do the heavy lifting asynchronously.

**Why not `@nestjs/microservices`?**
`amqp-connection-manager` directly gives full control over queue declarations, Dead Letter Exchanges, prefetch, publisher confirms, and graceful shutdown semantics. `@nestjs/microservices` abstracts these away and makes failure modes harder to reason about.

**Why OpenResty instead of plain Nginx?**
OpenResty is Nginx + LuaJIT — used in production by Cloudflare. Lua scripts connect to Redis at the edge for distributed rate limiting (INCR + expire), JWT revocation blacklist checks, and auth result caching (30s TTL). Plain Nginx cannot do this without a sidecar service and an extra network hop per request.

**CAP Theorem tradeoffs:**
- *Place Bet* → **CP** (Consistency + Partition-tolerance). Money integrity is non-negotiable. If the Wallet Service is unavailable, bet placement returns an error rather than accepting a bet that might not be funded.
- *Live Odds* → **AP** (Availability + Partition-tolerance). Slightly stale odds are acceptable. A blank screen is not. The system serves the last known odds from Redis even when the Market Service is down.

**Money as integers:**
All monetary values are stored and calculated as cents (`$10.50` = `1050`). Decimal odds are stored as integers ×100 (`2.40` = `240`). Settlement formula: `payout_cents = Math.floor(stake_cents * odds_int / 100)`. No floats anywhere in the financial path.

**Idempotency:**
Every bet request carries a `client_mutation_id` (UUID v4 generated by the client). The server stores `{user_id}:{client_mutation_id}` in Redis with a 60-second TTL. Network retries and double-taps return the original cached response without re-processing.

## Development

```bash
make up              # Start all infrastructure
make down            # Stop all infrastructure
make down-v          # Stop and wipe volumes (clean slate)
make logs            # Tail all container logs
make logs svc=openresty   # Tail a specific service
make ps              # Container status
make certs           # Regenerate self-signed TLS certificates

pnpm install         # Install dependencies
pnpm start:dev       # Start app in watch mode
pnpm test            # Run unit tests
pnpm lint            # Lint + auto-fix
```
